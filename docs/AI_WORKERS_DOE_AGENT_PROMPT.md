# Agent Prompt: Implement DOE Framework for AI Workers

You are implementing the Directive-Orchestration-Execution (DOE) framework for the AI Workers system in the OnCallShift codebase. This will make AI workers more reliable by separating concerns: directives (SOPs), orchestration (AI decision-making), and execution (deterministic scripts).

## Important Context

- **Repository**: OnCallShift - an incident management platform
- **Current State**: AI Workers already exist and work. They run Claude Code in ECS Fargate containers triggered by Jira issues labeled `ai-worker`.
- **What We're Adding**: DOE framework structure to make workers more reliable
- **Manager Review**: Disabled by default. The `ai-worker-manager.ts` Lambda exists but should be skipped unless the task explicitly requests review.

## The DOE Philosophy

> "90% accuracy per step = 59% success over 5 steps. Push complexity into deterministic code."

The AI agent (Claude Code) should:
1. **Read directives** from `directives/` to understand HOW to do things
2. **Call execution scripts** from `execution/` to DO things
3. **Self-anneal when things break** - fix the script, test it, update the directive
4. **NOT write code directly** when a script exists - use the tools

### Task 3: Create Execution Scripts

Each script should:
- Be a standalone TypeScript file
- Accept inputs via environment variables or stdin
- Output JSON to stdout with `{ success: boolean, ... }`
- Handle errors gracefully and return structured error info
- Be testable in isolation

**Example structure for `execution/git/create_pr.ts`:**

```typescript
#!/usr/bin/env npx ts-node

/**
 * Creates a GitHub PR for the current branch
 *
 * Inputs (environment variables):
 * - JIRA_KEY: Issue key for PR title
 * - JIRA_SUMMARY: Issue summary for PR title
 * - DESCRIPTION: PR body content
 * - GITHUB_TOKEN: Auth token
 *
 * Outputs (JSON to stdout):
 * - success: boolean
 * - prUrl: string (if success)
 * - prNumber: number (if success)
 * - error: string (if failure)
 */

import { execSync } from 'child_process';

interface Output {
  success: boolean;
  prUrl?: string;
  prNumber?: number;
  error?: string;
}

async function main(): Promise<void> {
  const output: Output = { success: false };

  try {
    const jiraKey = process.env.JIRA_KEY;
    const jiraSummary = process.env.JIRA_SUMMARY;
    const description = process.env.DESCRIPTION || '';

    if (!jiraKey || !jiraSummary) {
      throw new Error('JIRA_KEY and JIRA_SUMMARY are required');
    }

    const title = `${jiraKey}: ${jiraSummary}`;
    const body = `## Summary\n\n${description}\n\n---\n🤖 Generated by AI Worker`;

    // Get current branch and push
    const branch = execSync('git branch --show-current', { encoding: 'utf-8' }).trim();
    execSync(`git push -u origin ${branch}`, { stdio: 'pipe' });

    // Create PR
    const result = execSync(
      `gh pr create --title "${title.replace(/"/g, '\\"')}" --body "${body.replace(/"/g, '\\"')}" --json url,number`,
      { encoding: 'utf-8' }
    );

    const pr = JSON.parse(result);
    output.success = true;
    output.prUrl = pr.url;
    output.prNumber = pr.number;

  } catch (error: any) {
    output.success = false;
    output.error = error.message;
  }

  console.log(JSON.stringify(output));
  process.exit(output.success ? 0 : 1);
}

main();
```

**Priority scripts to create:**

5. `test/run_typecheck.ts` - Run `npx tsc --noEmit`

### Task 4: Create AGENTS.md

Create `backend/ai-worker/AGENTS.md` with instructions for Claude:

```markdown
# Agent Instructions

> This file provides context to Claude Code when executing AI Worker tasks.

You operate within a 3-layer architecture that separates concerns:

## Layer 1: Directive (What to do)
- SOPs in `directives/` defining goals, steps, edge cases
- Read the relevant directive FIRST before doing anything
- Directives are organized by persona: `backend_developer/`, `frontend_developer/`, etc.

## Layer 2: Orchestration (You)
- Your job: read directives, call execution scripts, handle errors
- You're the decision-maker, not the implementer
- DO NOT write shell commands directly - use scripts from `execution/`

## Layer 3: Execution (Tools)
- Deterministic TypeScript scripts in `execution/`
- Call these instead of running commands yourself
- Scripts output JSON so you can parse results

## Self-Annealing Protocol

When an execution script fails:

1. **Read the error** - Understand what went wrong
2. **Fix the script** - Modify the code in `execution/`
3. **Test it works** - Run the script again with same inputs
4. **Update the directive** - Add what you learned to the "Self-Annealing Notes" section
5. **Continue** - The system is now stronger

**Important:** Self-annealing improvements should be committed to a separate branch (`self-anneal/*`) and create a PR for human review.

## Key Principle

> "90% accuracy per step = 59% success over 5 steps. Push complexity into deterministic code."

When you find yourself writing the same command multiple times, that's a sign it should be an execution script.

## Current Task

Your task details are provided in the environment:
- JIRA_ISSUE_KEY: The Jira issue you're working on
- JIRA_SUMMARY: Issue summary
- JIRA_DESCRIPTION: Full issue description
- WORKER_PERSONA: Your role (backend_developer, frontend_developer, etc.)

Start by:
1. Reading `directives/common/git_workflow.md` to understand the PR process
2. Finding the directive that matches your task type
3. Following the directive step by step
```
